# Навчальний посібник: як влаштована `pip`

> Аудиторія: ті, хто вже знає базовий Python (класи, функції, винятки, модулі) і
> хоче розібратися в *архітектурі* маленької, але повноцінної програми — і в
> тому, як акуратно вбудувати в неї LLM. Це не туторіал із синтаксису, а розбір
> рішень: чому код поділено саме так і що з цього можна винести у власні проєкти.

`pip` — це чат-тамагочі в терміналі. Маленьке створіння має потреби (голод,
енергія, настрій), які спадають у реальному часі; воно саме подає голос, коли
йому погано, і ти з ним розмовляєш та дбаєш про нього в одному чаті. Голос —
модель Anthropic (Haiku); без API-ключа вмикаються прості templated-репліки, і
воно все одно живе.

Уся програма — близько 300 рядків чистого Python зі стандартної бібліотеки плюс
*опційний* пакет `anthropic`. Саме тому її зручно читати цілком.

---

## 1. Головна ідея: два годинники

Запам'ятай одну метафору — і вся решта стане на місце.

```
   ┌─────────────────────────┐        ┌──────────────────────────┐
   │   ТІЛО  (needs.py)       │        │   РОЗУМ  (brain.py)       │
   │                         │        │                          │
   │  голод/енергія/настрій  │        │  голос + дрібні рішення   │
   │  спадають у реальному   │        │  templated АБО LLM        │
   │  часі — чиста математика│        │  питаємо лише коли є      │
   │  завжди працює          │        │  що сказати               │
   └───────────┬─────────────┘        └────────────┬─────────────┘
               │   швидкий годинник                 │  повільний годинник
               │   (тікає сам)                       │  (за подією)
               └──────────────┬──────────────────────┘
                              ▼
                    ┌──────────────────┐
                    │   Pet (pet.py)    │  ← єдине місце, де вони зустрічаються
                    └──────────────────┘
```

- **Тіло** тікає **постійно і детерміновано**. Відійдеш на хвилину —
  повернешся до зголоднілого створіння. Це чиста математика: жодного LLM,
  жодного вводу-виводу, нічого, що може «зависнути» чи коштувати грошей.
- **Розум** питають **лише за подією** — коли ти щось написав чи погодував. Це
  єдине місце, де може бути мережа, затримка чи помилка.

Чому це важливо? Бо повільне, дороге й ненадійне (виклик LLM) ми чітко
відокремили від швидкого, дешевого й завжди справного (математика потреб). Це
той самий принцип, що в «дорослих» симуляціях світу, тільки доведений до суті:
**прибрано світ, лишилося тіло + потреби + LLM як голос.**

## 2. Карта пакета

`pip.py` у корені — тонкий лаунчер. Уся логіка в пакеті `creature/`, де **кожен
файл — одна ідея**:

| Файл | Роль | Залежить від |
|---|---|---|
| `creature/needs.py` | **ТІЛО** — потреби та їхній спад | нічого (чиста математика) |
| `creature/render.py` | мордочка, смужки, рядок статусу | `needs` (лише для типів) |
| `creature/brain.py` | **РОЗУМ** — `RuleBrain` і `LLMBrain` | `needs`, `config`, опц. `anthropic` |
| `creature/config.py` | конфіг LLM з `.env` | стандартна бібліотека |
| `creature/pet.py` | оркестрація: тіло + розум | `needs`, `brain` |
| `creature/chat.py` | цикл реального часу | `pet`, `render` |
| `creature/cli.py` | прапорці й запуск | `brain`, `pet`, `chat` |

Зверни увагу на стовпчик «залежить від»: стрілки залежностей течуть **в один
бік** — від циклу до тіла, ніколи навпаки. `needs.py` не знає про існування
`brain.py`, а `brain.py` не знає про термінал. Це і є *розділення
відповідальностей* (separation of concerns) на практиці, і саме воно дає змогу
тестувати тіло без мережі, а голос — без термінала.

> Чому пакет називається `creature`, а не `pip`? Бо каталог `pip/` затінив би
> справжній встановлювач пакетів `pip`. Створіння *за замовчуванням* зветься
> «pip», але код живе в нейтрально названому пакеті.

## 3. Розбір по модулях

### 3.1 `needs.py` — тіло

Серце — клас-`dataclass` із трьох чисел у діапазоні `[0, 1]`, де `1` завжди
«добре»:

```python
@dataclass
class Needs:
    hunger: float = 0.8  # ситість (fullness)
    energy: float = 0.9
    mood: float = 0.7
```

**Деталь, яку легко пропустити:** `hunger` зберігає *ситість*, а не голод. Тобто
`1.0` = повністю наїлося, `0.0` = вмирає з голоду. Завдяки цьому всі три потреби
підкоряються одному правилу «більше = краще», і рендеринг та пороги можуть
поводитися з ними однаково, без винятків. Маленьке рішення про знак — велике
спрощення всього іншого коду.

Спад керується одним словником-константою:

```python
DRAIN_PER_SECOND = {"hunger": 0.008, "energy": 0.005, "mood": 0.004}

def advance(self, dt, speed=1.0):
    for need, rate in DRAIN_PER_SECOND.items():
        drained = getattr(self, need) - rate * dt * speed
        setattr(self, need, max(0.0, drained))
```

Тут варто зупинитися на двох речах:

1. **`dt` — це реальний час.** Метод не «віднімає по чуть-чуть щотіку»; йому
   передають, скільки *секунд* реально минуло, і він масштабує спад на `dt`.
   Завдяки цьому швидкість спаду не залежить від того, як часто крутиться цикл.
   (Звідки береться `dt` — побачимо в `chat.py`.)
2. **Цикл по словнику замість трьох копій.** Імена потреб — це ключі
   `DRAIN_PER_SECOND`, тому `advance`, `levels`, `most_urgent` ітеруються по
   ньому й не дублюють логіку. Додаси четверту потребу — міняти треба лише
   словник і поля класу.

Решта методів — суто читання стану: `descriptors()` перетворює числа на слова
(«very hungry» / «getting hungry» / «well fed») для промпта LLM, `most_urgent()`
повертає назву найнижчої потреби, `clamp()` повертає значення в `[0, 1]` після
того, як годування підкинуло число вище одиниці.

> **Урок:** тримай «двигун» гри чистою функцією стану та часу. Жодних `print`,
> жодних запитів — тоді його тривіально перевіряти: `advance(10)` має зменшити
> голод рівно на `0.008 * 10`. Саме такий рядок є в смоук-тесті.

### 3.2 `render.py` — відображення

Цей модуль — **єдиний**, хто знає про ANSI-кольори й рамки. Усе тут — чисті
функції від `Needs`: дай стан — отримаєш рядок. Власного стану немає.

Ключова функція — `face_state`, і це гарний приклад **каскаду пріоритетів**:

```python
def face_state(n):
    if n.energy < 0.25: return "tired"   # фізичні потреби — найперше
    if n.hunger < 0.25: return "hungry"
    if n.mood   < 0.3:  return "sad"      # емоційні — потім
    if n.mood > 0.75 and n.hunger > 0.5 and n.energy > 0.5:
        return "happy"                    # ейфорія — лише коли все чудово
    if n.mood > 0.5 and n.hunger > 0.5 and n.energy > 0.5:
        return "content"                  # комфортно, але без захвату
    return "ok"
```

Порядок `if`-ів — це і є логіка: втома та голод «перебивають» поганий настрій, а
обличчя «happy» з'являється лише тоді, коли *всі* потреби в комфорті. `content` —
м'якший щабель між «happy» і «ok». Зміни порядок рядків — і зміниш характер
створіння, не торкаючись чисел.

`bar()` малює смужку `####----` і сам обирає колір (зелений/жовтий/червоний) за
рівнем; `status_line()` склеює бокс-обличчя зліва й три смужки справа. Якщо
колись захочеш інший «скін» (емодзі, інші рамки) — міняти треба **тільки цей
файл**.

**Мордочки в стилі Claude (v0.2).** Самі обличчя — у мінімалістичному, «сяйливому»
стилі (`✦ ✶ ✺ ◦` замість старого ASCII): `(✦‿✦)` щаслива, `(✶‿✶)` спокійна,
`(◦‿◦)` звичайна, `(✺o✺)` голодна, `(-_-)` втомлена, `(⌢_⌢)` сумна. Тут є
підступ, на який варто звернути увагу: **багато «сяйливих» гліфів у деяких
терміналах двоширинні** — і тоді рамка «попливе». Тому в `render.py` є маленька
сторожа на `unicodedata.east_asian_width`:

```python
def is_single_width(ch):
    # Na/N/H — надійно одна колонка; W/F — широкі; A — двоширинні в CJK-локалі
    return unicodedata.east_asian_width(ch) in {"Na", "N", "H"}
```

`_assert_faces_single_width()` проганяє **кожен** гліф облич через неї **на
імпорті** — отже, додаси випадково двоширинний символ, і програма впаде гучно й
одразу, а не намалює криву рамку. (Сама рамка `╭──╮` формально «ambiguous», але
вона незмінна й давно вирівнюється на практиці, тож її ця перевірка не чіпає.)

**Стан «думає» (seam для v0.4).** Окрім облич від потреб є ще `thinking` `(✻⋅✻)` —
це *тимчасовий стан інтерфейсу* (летить відповідь), а не похідне від тіла, тож
`face_state` його **ніколи не повертає**. Показати його можна лише через явний
перевизначник: `status_line(name, n, state="thinking")` (так само `face_block` і
`face`). Без аргументу — поведінка незмінна, тому старі виклики не зачеплені. Це
саме той **шов**, який v0.4 смикне, поки асинхронний мозок готує репліку.

**Обрамлений UI (v0.3).** Тут же живуть прості примітиви рамок: `box(lines, width,
*, title=None, color=None)` малює рамку рівно завширшки `width` (контент
доповнюється пробілами або обрізається — рамка ніколи не «пливе»), а
`columns(blocks)` ставить кілька рамок поруч, вирівнюючи рядки. З них зібрано
`welcome_banner(name, width, n)` — вітальний банер у дві колонки (привітання |
команди). Ключовий принцип той самий, що й скрізь у `render.py`: **функції
лишаються чистими — ширину їм передають ззовні**. Сам термінал міряє вже `chat.py`
(`shutil.get_terminal_size`), тримаючи весь ввід-вивід на «краю». Зверни увагу на
маленьку хитрість `_visible_len`: вона рахує ширину рядка, ігноруючи ANSI-коди, —
щоб кольорові рамки все одно вирівнювалися по колонках.

### 3.3 `brain.py` — розум

Тут найбільше архітектурних ідей. Почнемо з **контракту** — того, що пов'язує
обидва мозки:

```python
class Brain(Protocol):
    label: str
    def respond(self, name, kind, needs, user_text, history) -> Reply: ...
    def spontaneous(self, name, needs) -> str | None: ...
```

`Protocol` (з `typing`) описує інтерфейс **без успадкування**: будь-який клас із
цими методами «є» `Brain` для перевірки типів. Решта програми працює з абстракцією
`Brain` і ніколи не питає «а ти rule чи llm?». Це *поліморфізм через duck typing*,
зафіксований у типах.

Кожна відповідь має той самий крихітний形 — `{"say": str, "mood": float}`:
рядок, який сказати, і наскільки цей момент зрушив настрій (`+` за ласку й гру,
`−` за ігнорування). Цей формат — спільна мова між мозком і тілом, тож він
визначений в одному місці.

**Два мозки, один інтерфейс:**

- `RuleBrain` — словник готових реплік. Без мережі, миттєвий, завжди доступний.
- `LLMBrain` — справжня розмова через Anthropic. Усередині — той самий
  `RuleBrain` як **запасний** (`self.fallback`).

Тепер найважливіше — **дві навмисні «м'які» деградації** (graceful degradation),
які роблять створіння невбивним:

```python
def respond(self, name, kind, needs, user_text, history):
    try:
        ... # виклик API
        return self._parse(text)
    except Exception as exc:        # будь-яка біда: немає ключа, мережа, кривий JSON
        self.last_error = type(exc).__name__
        return self.fallback.respond(name, kind, needs, user_text, history)
```

1. **Помилка API → templated-репліка.** Виклик загорнуто в `try/except`, і
   *будь-який* збій повертає відповідь `RuleBrain`. Створіння ніколи не падає й
   не зависає через мережу.
2. **Спонтанні нагадування завжди templated:**

   ```python
   def spontaneous(self, name, needs):
       return self.fallback.spontaneous(name, needs)  # навмисно не через API
   ```

   Фонові «...i'm getting hungry» — миттєві й безкоштовні. LLM питаємо **лише**
   за прямою дією користувача чи реплікою, ніколи — для фонового бурмотіння. Це
   повертає нас до «двох годинників»: повільний годинник не має права тікати сам
   по собі.

Як виглядає інтеграція LLM по суті — три кроки:

- **`_build_context`** пакує стан тіла й «що щойно сталося» в JSON. Тобто промпт
  отримує і `feeling` (слова), і `levels` (числа), і останні 4 репліки —
  достатньо, щоб відповісти «в характері та про самопочуття».
- **`SYSTEM`** — особистість і, головне, **жорсткий контракт виводу**: «відповідай
  ЛИШЕ JSON-об'єктом `{"say": ..., "mood": ...}`».
- **`_parse`** читає відповідь *толерантно*: зриває ```` ```json ````-огорожі й,
  якщо JSON усе одно кривий, повертає сирий текст із `mood=0.0`. Модель іноді не
  слухається — код це переживає, а не падає.

> **Урок про LLM-у застосунку:** стався до моделі як до ненадійного зовнішнього
> сервісу. Дай чіткий контракт виводу, **парси оборонно**, май запасний шлях — і
> ніколи не клади на критичний шлях те, що можна зробити дешево й локально.

### 3.4 `config.py` — конфіг із `.env`

Усі «ручки» голосу зібрані в одному місці: яка модель, яка довжина відповіді, де
ключ. Значення беруться з оточення, а локальний (і прихований від git) файл
`.env` підвантажується в оточення на імпорті:

```python
MODEL = os.environ.get("PIP_MODEL", DEFAULT_MODEL)
MAX_TOKENS = int(os.environ.get("PIP_MAX_TOKENS", "120"))
```

Парсер `.env` — крихітний і без сторонніх залежностей (ніякого `python-dotenv`),
і має дві важливі властивості:

- **не перезаписує** змінну, яка вже є в реальному оточенні (тобто `export
  ANTHROPIC_API_KEY=...` завжди має перевагу над файлом);
- **порожнє значення** в `.env` — це «не задано», тож заготовка-плейсхолдер
  нічого не ламає.

`SYSTEM`-промпт навмисно залишився в `brain.py`: це *особистість*, а не
налаштування. Хороше правило: у `.env` — секрети та прості «ручки» (ключ,
модель, ліміти), у коді — логіка та характер.

> **Урок:** ключ не місце ані в коді, ані в історії git. Шаблон `.env` +
> `.env.example` + рядок у `.gitignore` — стандартний спосіб тримати секрети
> поза репозиторієм, але поряд із кодом.

### 3.5 `pet.py` — оркестрація

`Pet` — єдине місце, де тіло й розум зустрічаються, і він навмисно **тонкий**.
Сам нічого не рахує — лише маршрутизує час і події між половинами.

Ефекти турботи винесено в **таблицю-дані**, а не сховано в `if`-ах:

```python
ACTION_EFFECTS = {
    "feed":  {"hunger": +0.45, "mood": +0.05, "energy": +0.05},
    "play":  {"mood": +0.35, "energy": -0.20, "hunger": -0.10},
    "sleep": {"energy": +0.60, "mood": +0.05},
}

def act(self, kind):
    for need, delta in ACTION_EFFECTS.get(kind, {}).items():
        setattr(self.needs, need, getattr(self.needs, need) + delta)
    self.needs.clamp()
```

Правила гри тепер можна прочитати очима за секунду й змінити, не чіпаючи логіку.
Зверни увагу: гра підіймає настрій, але **виснажує** енергію й трохи голод — це
видно прямо в таблиці.

Другий цікавий метод — `respond`: він не лише питає мозок, а й **застосовує
повернутий зсув настрою назад на тіло**. Ось де замикається петля «розум впливає
на тіло»:

```python
reply = self.brain.respond(...)
self.needs.mood = clamp(self.needs.mood + reply.get("mood", 0.0))
```

`spontaneous` тримає «остуду» (cooldown), щоб створіння не нило щосекунди, і
масштабує її на `speed`.

### 3.6 `chat.py` — цикл реального часу

Це диригент. Один прохід циклу:

```python
while True:
    now = time.time()
    pet.tick(now - last)   # 1. тіло тікає на реально минулий час
    last = now

    spont = pet.spontaneous(now)   # 2. може, саме подати голос
    if spont: ...

    # 3. НЕблокувальне читання: читаємо лише якщо у stdin щось є
    if select.select([sys.stdin], [], [], 0.2)[0]:
        line = sys.stdin.readline()
        ...
```

Тут — два прийоми, заради яких усе працює:

1. **`now - last` дає `dt`.** Замість «спати рівно секунду» ми міряємо, скільки
   реально минуло між проходами, і передаємо це в тіло. Тому навіть якщо ти
   надовго задумався над відповіддю, голод нарахується правильно.
2. **`select.select(..., 0.2)` — неблокувальний ввід.** Звичайний `input()`
   *заморозив* би програму, доки ти не натиснеш Enter — і час би не йшов.
   Натомість `select` чекає на ввід щонайбільше 0.2 с; якщо нічого нема —
   повертаємось у цикл, тікаємо далі й знову подаємо голос за потреби. Саме це
   робить час «справжнім».

> ⚠️ `select` на `stdin` — це Unix (macOS/Linux). На Windows цей шматок не
> працює; це свідомий компроміс заради простоти.

Команди (`/feed`, `/status`, ...) розбирає `_handle`; будь-який інший текст стає
розмовною реплікою (`kind="chat"`). `_handle` повертає лише прапорець «виходимо?»;
репліки друкує одразу.

#### Статус на вимогу (і чому не «живий футер»)

Мордочку й смужки показує `_show_status` — **на вимогу**: одразу під банером, після
догляду й за `/status`. Більше нічого: розмова прокручується звичайним терміналом,
а спонтанні репліки просто друкуються рядком. **Жодного керування курсором.**

Чому не зробити «живий футер», прибитий унизу, що сам оновлюється? Бо це чесна
межа можливого. Раніше (v0.1–v0.3) ми так і робили: блок статусу перемальовувався
на місці курсором (`cursor_up` + `clear_below`, `save`/`restore`). Виглядало гарно
— смужки повзли самі. Але такий підхід **рахує рядки терміналу й рухає курсор**, а
це крихко: щойно репліка переноситься на новий рядок, emoji розширює рядок або
спонтанна репліка втручається між натисканнями — арифметика «з'їжджає» й лишає
**напівнамальовані рамки**. Я кілька разів намагався це «полатати» — і це була
помилка.

> **Урок:** живий, прибитий статус у звичайному терміналі надійно не зробити — це
> якраз той випадок, коли треба **захопити екран** (повноекранний перемальовок,
> alternate screen). Це і є **клієнт демона у v1.1**. Доти найнадійніше — статус
> на вимогу: нуль escape-кодів, нічого не ламається, а піднаправлений вивід чистий
> сам собою. (Примітиви курсора лишилися в `render.py` саме під той майбутній
> клієнт.) Відображення й далі живе «на краю»: ядро (`needs`, `pet`,
> рендер-функції) не чіпаємо; `chat.py` лише друкує готові рядки.

### 3.7 `cli.py` — складання докупи

Найкоротший модуль, і це показово. Він лише читає прапорці й **з'єднує деталі**:

```python
brain = make_brain(args.name, prefer_llm=not args.rule)
pet = Pet(args.name, brain, speed=args.speed)
run(pet)
```

Вибір мозку (`make_brain`) ізольований тут, на «краю» програми. Серцевина
(`Pet`, `Needs`, `run`) не знає й не хоче знати, який мозок їй дали — вона працює
з контрактом `Brain`. Це називають *dependency injection*: залежність створюють
зовні й «вкидають» усередину, замість того щоб серцевина сама її добувала.

## 4. Потік даних: що відбувається, коли ти пишеш «hello»

Простежмо один такт наскрізь — це найкращий спосіб побачити, як частини
співпрацюють:

```
ти набираєш «hello» ↵
        │
        ▼
chat.run() ── select помітив ввід ──► _handle(pet, "hello")
        │
        ▼  (не команда → розмова)
pet.respond("chat", "hello")
        │
        ├─► brain.respond(...)              # РОЗУМ
        │        │
        │        ├─ LLMBrain: _build_context → виклик Haiku → _parse
        │        │     └─ якщо збій → fallback RuleBrain
        │        └─ повертає {"say": "...", "mood": +0.05}
        │
        ├─► needs.mood += 0.05              # зсув настрою лягає на ТІЛО
        ├─► history.append(...)             # пам'ять (останні 8 реплік)
        │
        ▼
chat друкує репліку (а статус — на вимогу: /status чи після догляду)   # ВІДОБРАЖЕННЯ
        │
        ▼
цикл крутиться далі — тіло тікає, час іде
```

Зверни увагу, як рівно розкладені обов'язки: розум **сказав**, тіло **змінилось**,
рендер **показав** — і жоден із них не знає подробиць іншого.

## 5. Чотири уроки, які варто забрати з собою

1. **Дві швидкості — два модулі.** Відокрем те, що мусить бути швидким, дешевим і
   надійним (математика стану), від того, що повільне, дороге й ненадійне (виклик
   моделі). Не давай повільному тікати на критичному шляху.
2. **М'яка деградація замість падінь.** Немає ключа? Впав запит? Кривий JSON?
   Завжди є простіший, локальний запасний шлях. Користувач не повинен бачити
   стектрейс там, де можна сказати «mmm, that's better».
3. **LLM — це ненадійний компонент, а не магія.** Чіткий контракт виводу в
   промпті, оборонний парсинг відповіді, fallback. Стався до моделі як до
   мережевого сервісу, що може відповісти будь-що.
4. **Дані замість коду там, де це правила.** Швидкості спаду, ефекти дій, обличчя
   — усе це таблиці/словники, які читаються за секунду. Логіка лишається тонкою,
   а «балансування гри» стає редагуванням даних.

## 6. Запусти й поекспериментуй

```bash
python pip.py                 # потрібен ключ у .env або в оточенні
python -m creature            # те саме, інший вхід
python pip.py --rule          # лише templated-голос, без API
python pip.py --speed 8       # потреби спадають швидко — зручно для дослідів
python pip.py --name moss     # інше ім'я
```

Перевірити цикл без рук (передати команди через конвеєр):

```bash
printf '/status\nhello\n/feed\n/play\n/quit\n' | python pip.py --rule --speed 5
```

   `Needs`, один рядок у `DRAIN_PER_SECOND`, гілка в `descriptors`, команда
   `/wash` в `ACTION_EFFECTS` і `_handle`. Скільки місць довелося чіпати — це

2. **Новий вираз обличчя.** Додай стан `"sick"` (хворе), коли дві потреби нижчі
   за 0.2 одночасно. Зміни лише `render.py` — і переконайся, що нові гліфи
   проходять перевірку `is_single_width` (інакше `_assert_faces_single_width`
   завалить імпорт), щоб рамка лишилася рівною.
3. **Пам'ять між сесіями.** Зберігай `pet.needs` і `pet.history` у JSON при
   виході й завантажуй на старті. Де це найчистіше вкрутити, не «забруднивши»
   тіло й розум? (Підказка: на «краю» — у `cli.py`/`chat.py`.)
4. **Температура як ручка.** Додай `PIP_TEMPERATURE` у `config.py` і `.env.example`,
   передай її у виклик `messages.create`. Поспостерігай, як змінюється «характер»
   реплік.
5. **Двошаровий голос.** Зараз спонтанні репліки — завжди templated. Зроби так,
   щоб *зрідка* (з малою ймовірністю) спонтанна репліка йшла через LLM. Де тут
   ризик для «дешевого фонового голосу» і як його обмежити?
6. **Тести.** Напиши кілька `assert`-тестів на `Needs.advance`, `Pet.act` і
   `LLMBrain._parse` (останній — на кривому вводі). Зверни увагу, що тіло й
   парсер тестуються **без жодної мережі** — це і є винагорода за чисте
   розділення.

> годинників* належить твоя зміна, і в який модуль вона має лягти. Якщо відповідь
> очевидна — архітектура працює.
