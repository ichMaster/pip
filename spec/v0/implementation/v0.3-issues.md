# Phase v0.3 — Framed UI redesign

> Source of truth: the **v0.3** section of [spec/ROADMAP.md](../../ROADMAP.md).
> This file is the fine-grained breakdown consumed by `/upload-issues`.

**Goal:** a cleaner, Claude-Code-adjacent presentation — a **width-aware welcome
banner** and **framed, colored panels** — while keeping the natural-scrollback
chat model (no full-screen takeover).

Today the UI is a face block + three needs bars
([`status_line`](../../../creature/render.py)) drawn as an in-place footer (v0.1).
There is no welcome banner, no boxed/columned panels, and the layout is a fixed
width regardless of terminal size. The intro is three bare `print`s in
[`chat.py`](../../../creature/chat.py) (`blinks awake` + `HELP` + `hi!`).

**Scope & guardrails (from the roadmap and [CLAUDE.md](../../../CLAUDE.md)):**
- **Pure render functions** stay in [`render.py`](../../../creature/render.py) (the
  sole ANSI owner). The framing helpers take an explicit **`width`** and return
  strings — they do **no I/O**. The `shutil.get_terminal_size()` call lives in
  [`chat.py`](../../../creature/chat.py) (the placement layer), which passes width in.
  This mirrors v0.1: chat.py owns *placement / measurement*, render.py owns *strings*.
- **No new dependency** (`shutil` is stdlib). Glyphs stay **single-width** — the
  import-time `_assert_faces_single_width` still guards face glyphs; the box frame
  (`╭─╮│╰╯`) is exempt as before. ANSI still degrades where ignored.
- **Preserve the scrollback chat model.** Conversation scrolls in the normal
  terminal; the status stays an **in-place footer**. **No `curses` / alternate-screen
  TUI** — a full-screen frame is the `v1.1` / `v2.6` client, not v0.
- **Footer behavior is unchanged** — the v0.1 in-place redraw, no-stacking, and
  scrollback preservation all still hold. The footer may become *width-aware*
  (bars fit the terminal), but its **redraw behavior and 4-line height stay the
  same** so the v0.1 cursor arithmetic (`STATUS_HEIGHT`) is untouched.
- **Two-clock split intact:** body ([`needs.py`](../../../creature/needs.py)),
  [`pet.py`](../../../creature/pet.py), and the brains are not touched.
- **Docs ship with code.**
- **Release:** completing this phase cuts `0.3.0` (via `/release-version`, user's call).
- **Forward link:** visual groundwork for `v1.1`'s live full-frame client and
  `v2.6`'s framed fleet TUI — raw-ANSI and scrollback-preserving here.

## PIP-008: Width-aware framing primitives (box / columns)

### Description
Add the reusable, **pure** building blocks for the framed UI: draw a bordered box
of a given width, and place boxed panels side by side. Everything the welcome
banner and any panel needs, as width-parameterised functions in `render.py`.

### What needs to be done
- In [`render.py`](../../../creature/render.py), add pure helpers (no I/O; take
  `width`, return `list[str]` or `str`):
  - `box(lines, width, *, title=None, color=None)` — wrap content `lines` in a
    bordered box exactly `width` columns wide (reuse the existing `╭ ─ ╮ │ ╰ ╯`),
    padding each line to the inner width and **truncating** lines that overflow
    (so the frame never smears). Optional `title` sits in the top border; optional
    `color` wraps the frame.
  - `columns(blocks, *, gap=1)` — place several equal-height boxed `blocks`
    (each a `list[str]`) side by side with `gap` spaces between, returning the
    combined `list[str]`. Pad shorter blocks so rows align.
  - A small `_pad`/`_fit(text, width)` helper that pads-or-truncates a single line
    to an exact column count (accounting for the leading box padding).
- Keep glyphs single-width; the box frame chars are exempt from the face guard as
  today. Degrade gracefully (a box is still readable if ANSI is stripped).
- **No terminal measurement here** — these take `width` as an argument; chat.py
  supplies it (PIP-009/010).

### Dependencies
None.

### Expected result
`render.py` exposes `box(...)` and `columns(...)` (plus a line-fit helper) that
produce aligned, width-exact framed output, ready for the welcome banner and any
panel. No behavior change on their own.

### Acceptance criteria
- [ ] `box(lines, width)` returns lines that are **all exactly `width` columns**, with
      a correct `╭─╮ / │ / ╰─╯` border; over-long content is truncated, short content padded.
- [ ] `box(..., title=...)` renders the title in the top border without breaking width.
- [ ] `columns([a, b])` aligns two boxes side by side (equal rows, `gap` between),
      every combined row equal width.
- [ ] Helpers are **pure** (take `width`, return strings, no I/O); stdlib-only.
- [ ] Single-width invariant intact (`_assert_faces_single_width` still passes);
      box frame stays aligned.
- [ ] `python -m py_compile creature/render.py` + import pass; existing
      `status_line` / faces unchanged.

## PIP-009: Welcome banner on startup

### Description
Replace the three bare intro `print`s with a **framed, colored welcome banner** —
a two-column panel (greeting | quick-help) styled close to Claude Code's framed
panels — drawn width-aware at startup.

### What needs to be done
- In [`render.py`](../../../creature/render.py), add a pure
  `welcome_banner(name, width, needs)` (or similar) that builds the banner from the
  PIP-008 primitives: a left panel (greeting — the compact `face` + "hi! i'm so glad
  you're here.") and a right panel (quick-help — the `/feed /play /sleep`,
  `/status /help /quit`, "anything else" lines), combined via `columns(...)` to fit
  `width`. Colored frame, on-brand.
- In [`chat.py`](../../../creature/chat.py), measure the terminal with
  `shutil.get_terminal_size()` and render the banner in place of the current
  `blinks awake` + `HELP` (+ optionally the `hi!` line, now inside the banner) at the
  top of `run()`.
- **Non-TTY / narrow degrade:** when output is not a TTY (the existing
  `_interactive()` check) or the width is too small for two columns, fall back to a
  plain-text greeting + help (no frame / single column) so piped runs and the
  smoke-test stay a clean plain log.
- Keep the banner a one-time startup element that **scrolls into history** (it is not
  the persistent footer); the footer still follows.

### Dependencies
- PIP-008 (`box` / `columns`).

### Expected result
Launching pip shows a framed, colored, two-column welcome banner sized to the
terminal; piped/narrow runs show a clean plain-text greeting + help instead.

### Acceptance criteria
- [ ] Startup shows a **framed, colored welcome banner** with a greeting panel and a
      quick-help panel, boxed and aligned, fitting the terminal width.
- [ ] The banner is built from the PIP-008 primitives (no ad-hoc box code in chat.py).
- [ ] `shutil.get_terminal_size()` is called in `chat.py` (render stays pure / width-fed).
- [ ] Non-TTY (piped) and too-narrow terminals degrade to a plain greeting + help;
      the smoke-test output stays a clean plain log (no escape soup, no smeared frame).
- [ ] The conversation + in-place footer still follow the banner (v0.1 behavior intact).
- [ ] `python -m py_compile` + import + smoke-test (exit 0) pass; `--rule` and LLM
      paths both show the banner.

## PIP-010: Width-aware footer + reflow on resize

### Description
Make the persistent UI **width-aware**: recompute the terminal width on each redraw
so the status footer (and any framed element) fits the terminal and **reflows when
the window is resized**, without changing the footer's in-place redraw behavior.

### What needs to be done
- In [`chat.py`](../../../creature/chat.py), recompute `shutil.get_terminal_size()`
  on each draw/redraw (intro, `_reframe`, idle `_refresh_status`) and thread the
  width into the render calls, so a resized terminal reflows on the next redraw.
- Make the footer width-aware in a **height-preserving** way: extend
  `status_line(name, n, state=None, width=None)` (and `bar`) so the needs bars /
  right column fit the available width (longer bars on wide terminals), while the
  block stays **4 lines** so `STATUS_HEIGHT` and the v0.1 cursor arithmetic are
  unchanged. `width=None` keeps today's fixed layout (back-compat).
- Clamp to a sensible **minimum width** and degrade on very narrow terminals (don't
  produce negative padding / smearing). Non-TTY keeps the plain-log path.
- Do **not** introduce `curses` / alternate-screen; this stays raw-ANSI and
  scrollback-preserving. Footer redraw/no-stacking behavior unchanged.

### Dependencies
- PIP-008, PIP-009 (the primitives and the width-measuring pattern in chat.py).

### Expected result
Resizing the terminal reflows the frame on the next redraw — boxes/bars fit the new
width with no smearing — while the footer keeps its v0.1 in-place behavior and height.

### Acceptance criteria
- [ ] Terminal width is recomputed on redraw; the status footer fits the width and
      **reflows when the terminal is resized** (verify wide vs narrow; no smearing).
- [ ] The footer stays **4 lines**; `STATUS_HEIGHT` and the v0.1 in-place redraw
      (no stacking, scrollback preserved, live idle refresh, cursor restore) are unchanged.
- [ ] Very narrow widths degrade gracefully (clamped, no negative padding / overflow).
- [ ] `status_line(..., width=None)` preserves the current fixed layout (back-compat).
- [ ] Non-TTY still degrades to a plain log; smoke-test exits 0 with no escape soup.
- [ ] No `curses` / alternate-screen; two-clock split intact (no needs/pet/brain change).
- [ ] `python -m py_compile` + import + smoke-test pass.
