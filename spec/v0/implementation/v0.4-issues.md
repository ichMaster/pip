# Phase v0.4 — Async brain (off-thread)

> Source of truth: the **v0.4** section of [spec/ROADMAP.md](../../ROADMAP.md).
> This file is the fine-grained breakdown consumed by `/upload-issues`.

**Goal:** the LLM reply no longer freezes the loop — the body keeps ticking and
input stays live during the ~1 s call, and pip shows it's thinking.

Today [`chat.py`](../../../creature/chat.py) calls `pet.respond("chat", …)`
**synchronously** inside `_handle`, and [`pet.respond`](../../../creature/pet.py)
calls `self.brain.respond(...)` inline, so the whole `run()` loop blocks for the
duration of the API call: the body clock stalls, input goes dead, and (with the
old footer) the screen could corrupt. There is **no `threading`/`queue`/`async`
anywhere** in `creature/` yet — this phase introduces it, in miniature.

**Reconciliation note (read before implementing):** the roadmap line *"show the
thinking face (v0.2) in the **stable picture (v0.1)**"* predates **v0.3**, which
**removed the live in-place footer** in favor of on-demand status (zero
cursor-control, can't corrupt). So there is no live picture to update. v0.4
therefore shows thinking as a **printed line** built from v0.2's `thinking` face
(`face(needs, state="thinking")` → `(✻⋅✻)`) — keeping the "no cursor choreography"
robustness from v0.3. PIP-015 updates the roadmap wording to match.

**Scope & guardrails (from the roadmap and [CLAUDE.md](../../../CLAUDE.md)):**
- **Single-thread body.** The body (`Needs`, `history`) is **mutated only on the
  main thread**. The worker gets an **immutable snapshot** and returns `{say, mood}`;
  the main thread applies the mood delta + appends history. (This is exactly the
  invariant `v1.2`'s daemon reuses — introduced here in miniature.)
- **Two-clock split intact.** The body keeps draining continuously; "thinking" is an
  *interaction* state, never a body state.
- **Preserve the degradations:** `LLMBrain.respond` still falls back to `RuleBrain`
  on any error (it runs on the worker, unchanged); **spontaneous nags stay templated
  and instant** (never via the API); **`--rule` stays effectively instant** and offline.
- **Stdlib only** (`threading` / `concurrent.futures` / `queue` are stdlib). No new dependency.
- **Robust I/O.** Keep v0.3's no-cursor-choreography rule: the thinking indicator is a
  printed line (or at most a **single-line** `\r` spinner — never multi-line cursor math).
- **Docs ship with code.**
- **Release:** completing this phase cuts `0.4.0` (via `/release-version`, user's call).
- **Forward link:** the single-process seed of `v1.2`'s daemon worker-pool async mind.

## PIP-012: Main-thread body contract — `snapshot` / `think` / `apply_reply`

### Description
Split [`Pet.respond`](../../../creature/pet.py) so the slow brain call can run on a
worker thread against a frozen snapshot, while every mutation of the body stays on
the main thread. No threading yet — this is the pure refactor that makes async safe.

### What needs to be done
- In [`pet.py`](../../../creature/pet.py), add three methods around the existing
  `respond`:
  - `snapshot()` → an immutable view the worker can read without races: a **copy of
    `Needs`** (e.g. `dataclasses.replace(self.needs)`) and a **copy of `history`**
    (`list(self.history)`). Return a small object/namedtuple (e.g. `(needs, history)`).
  - `think(kind, text, snap)` → **pure** (no body mutation): calls
    `self.brain.respond(self.name, kind, snap.needs, text, snap.history)` and returns
    the `{say, mood}` reply. Safe to run off the main thread (the brain only *reads*
    needs/history).
  - `apply_reply(kind, text, reply)` → **main thread only**: apply the mood delta to
    the live `self.needs` (clamped) and, for `kind == "chat"`, append
    `{you: text, name: say}` to `self.history` (trimmed to `HISTORY_LIMIT`). Returns `say`.
- Re-express `respond(kind, text)` as the thin synchronous composition
  `apply_reply(kind, text, think(kind, text, snapshot()))` so existing callers and
  `--rule` keep working unchanged.
- Document the **single-thread-body invariant** in the module docstring (the worker
  reads a snapshot; the main thread is the only writer).
- No I/O, no threads, no LLM knowledge added to `pet.py` (it still just routes).

### Dependencies
None.

### Expected result
`Pet` exposes `snapshot()` / `think()` / `apply_reply()`; `respond()` is their
synchronous composition. Behavior is identical to today, but the brain call is now
cleanly separable from body mutation.

### Acceptance criteria
- [ ] `snapshot()` returns copies (mutating the live body afterwards doesn't change a taken snapshot).
- [ ] `think()` performs **no** body mutation (needs/history unchanged by a call to it).
- [ ] `apply_reply()` applies the mood delta (clamped to `[0,1]`) and appends history only for `chat`.
- [ ] `respond()` == `apply_reply(... think(... snapshot()))` and is behavior-identical to before.
- [ ] Single-thread-body invariant documented; `pet.py` stays pure orchestration (no I/O/threads).
- [ ] `python -m py_compile creature/pet.py` + import pass; smoke-test exits 0; LLM→RuleBrain fallback unaffected.

## PIP-013: Off-thread brain execution (non-blocking loop)

### Description
Run the brain call on a background worker so `run()` keeps ticking the body and
polling input during the ~1 s API call; apply the result on the main thread when it
lands. This is the core async plumbing.

### What needs to be done
- In [`chat.py`](../../../creature/chat.py), introduce a single background worker
  (`concurrent.futures.ThreadPoolExecutor(max_workers=1)` **or** a `threading.Thread`
  + `queue.Queue`). On a chat/care turn: take `pet.snapshot()` on the main thread and
  submit `pet.think(kind, text, snap)` to the worker; **do not block**.
- Each loop pass, **poll** for a finished result (non-blocking: `future.done()` or a
  result `queue.Queue.get_nowait()`); when ready, call `pet.apply_reply(...)` **on the
  main thread** and print the reply line.
- The loop continues to `pet.tick(...)` and `select`-poll stdin while a reply is
  pending — the body never stalls.
- **`--rule` / offline:** the worker resolves almost immediately; the reply appears on
  the next poll. Keep it snappy (small poll interval is already `POLL_SECONDS`).
- **EOF / piped input:** before exiting on EOF, **drain any pending reply** (wait for
  the worker, apply it, print it) so no reply is lost — the documented smoke-test must
  still print pip's answers.
- **Ctrl-C / shutdown:** don't let the worker block exit (daemon thread, or
  `executor.shutdown(wait=False)` in a `finally`); `KeyboardInterrupt`/`EOFError` still
  exit cleanly (handled in `cli.main`).
- Keep the brain instance single-threaded in practice (one in-flight call at a time —
  PIP-014 enforces the queueing).

### Dependencies
- PIP-012 (`snapshot` / `think` / `apply_reply`).

### Expected result
Sending a chat line no longer freezes the program: bars keep moving and input stays
responsive during the call; the reply prints when the worker finishes.

### Acceptance criteria
- [ ] During a chat call the body keeps ticking and input stays responsive (no freeze) —
      verified with an artificially-slow brain (e.g. a `time.sleep` shim) in a PTY: bars
      advance while a reply is pending.
- [ ] The reply is applied on the **main thread** (`apply_reply`) and printed when ready.
- [ ] `--rule` and the no-API-key path still work and stay effectively instant.
- [ ] Piped EOF drains the pending reply before exit (smoke-test still prints answers; exit 0).
- [ ] `Ctrl-C` exits cleanly; no hung worker keeps the process alive.
- [ ] Body mutated only on the main thread (no `Needs`/`history` writes in the worker);
      LLM→RuleBrain fallback intact; stdlib-only.
- [ ] `python -m py_compile creature/chat.py` + import pass.

## PIP-014: Thinking indicator + interaction rules

### Description
Show that pip is thinking while a reply is pending, and make the interleavings the
async loop now allows well-defined — a small `IDLE → THINKING → IDLE` interaction
state layered over PIP-013.

### What needs to be done
- **Thinking indicator (robust, no cursor choreography).** When a brain call is
  submitted, print a thinking line built from v0.2's seam —
  `face(pet.needs, state="thinking")` → `(✻⋅✻)` — e.g. `  pip (✻⋅✻) …thinking`.
  When the reply lands, print the reply line. *(Optional polish: a **single-line** `\r`
  spinner on that one line — never `cursor_up`/multi-line. If unsure, keep it a static line.)*
- **Interaction state (`IDLE`/`THINKING`).** Track whether a brain call is in flight:
  - A **chat** sent while `THINKING` is **queued (FIFO)** and dispatched when the current
    reply resolves — never a second concurrent call.
  - A **care action** applies its **numeric effect immediately** (`pet.act` on the main
    thread, reprint status) in **any** state; its spoken line goes through the same
    async queue.
  - A **spontaneous** nag is **suppressed while `THINKING`** (don't interleave a nag into
    a pending exchange); it resumes once `IDLE`.
- Keep spontaneous **templated/instant** and `--rule` instant; preserve the
  LLM→RuleBrain fallback (it just runs on the worker).
- Stay within v0.3's robustness rule: everything is printed lines or a single-line
  spinner — **zero multi-line cursor arithmetic** (so it can't corrupt like the old footer).

### Dependencies
- PIP-013 (the off-thread execution + result polling).

### Expected result
While a reply is in flight pip shows `(✻⋅✻) …thinking`; a second chat queues instead of
launching a concurrent call; feeding mid-think still changes the bars immediately; a
spontaneous nag never interrupts a pending reply.

### Acceptance criteria
- [ ] While a reply is pending, a thinking indicator (`(✻⋅✻)`) is shown; it's replaced/
      followed by the reply when it lands.
- [ ] A chat sent mid-think **queues** (FIFO) — never a second concurrent brain call.
- [ ] A care action's **numeric effect applies immediately** in any state (bars change;
      `/status` reflects it); its spoken line goes async.
- [ ] A spontaneous nag is **suppressed while THINKING** and resumes when IDLE.
- [ ] Spontaneous stays templated/instant; `--rule` instant; LLM→RuleBrain fallback intact.
- [ ] No multi-line cursor control — output can't corrupt (verify zero `cursor_up`/`clear_below`
      in a spontaneous-storm PTY run); smoke-test exits 0.
- [ ] `python -m py_compile creature/chat.py` + import pass.
