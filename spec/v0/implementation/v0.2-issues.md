# Phase v0.2 — Claude-style face set

> Source of truth: the **v0.2** section of [spec/ROADMAP.md](../../ROADMAP.md).
> This file is the fine-grained breakdown consumed by `/upload-issues`.

**Goal:** replace the current ASCII faces with a richer set in a cleaner,
Claude-adjacent aesthetic (the `✻`/`✶` sparkle / minimal look), and add a
`thinking` face for v0.3's pending-reply state.

Today [`render.py`](../../../creature/render.py) has five faces
(happy / ok / hungry / tired / sad) built from ASCII eyes + mouth in
`FACE_PARTS` (boxed), `STATE_FACE` (compact one-liner) and `STATE_COLOR`, all
selected by the `face_state` priority cascade.

**Scope & guardrails (from the roadmap and [CLAUDE.md](../../../CLAUDE.md)):**
- This is **render-only work** in [`render.py`](../../../creature/render.py) (plus
  doc refreshes). `render.py` stays a set of **pure functions of a `Needs`** and the
  **sole owner of ANSI / box-drawing**.
- **Single-width glyphs only.** Many sparkle glyphs render *double-width* in some
  terminals (East Asian Width "Wide"/"Full"/"Ambiguous"), which would break the box.
  Width must be **verified programmatically**, not eyeballed — this is the central risk
  of the phase.
- **Keep the box dimensions** (7-wide interior, 4 lines) so `status_line` and
  `STATUS_HEIGHT` are unchanged and the **v0.1 in-place redraw keeps working**.
- **Stdlib only** (`unicodedata` is stdlib). No new dependency. Keep rendering
  ANSI-degrading.
- **Two-clock split intact:** the body ([`needs.py`](../../../creature/needs.py)),
  [`pet.py`](../../../creature/pet.py) and the brains are not touched. Rendering is
  brain-agnostic, so `--rule` and LLM paths render the same faces.
- **`thinking` is a UI state, not a need.** v0.2 only *provides* the face and the render
  seam; the live "show it while a reply is pending" wiring is **v0.3** (async brain).
- **Docs ship with code.**
- **Release:** completing this phase cuts `0.2.0` (via `/release-version`, user's call).

## PIP-005: Claude-style face set (width-safe)

### Description
Restyle the creature's faces into a cleaner, Claude-adjacent minimal / sparkle
aesthetic, keeping every glyph single-width so the box stays aligned. This is the heart
of the phase: a distinct, on-brand face set plus a programmatic width guard that makes
"sparkle glyphs are sometimes double-width" a caught error instead of a broken box.

### What needs to be done
- **Pick a Claude-adjacent glyph palette** for eyes + mouths (sparkle / minimal motif,
  e.g. `✶ ✦ ✧ ✺` eyes, soft `‿ ⌣ — ·` mouths). Keep the **same box** as today
  (`╭───────╮` … 7-wide interior, 4 lines) so `face_block` / `status_line` /
  `STATUS_HEIGHT` are unchanged; only the eyes/mouth strings change.
- **Add a width-safety guard** in `render.py`: a helper using
  `unicodedata.east_asian_width(ch)` that treats a glyph as box-safe **only** when its
  width is Narrow / Neutral / Halfwidth (i.e. reject `"W"`, `"F"`, and the
  context-dependent `"A"` ambiguous). Use it to **validate every glyph** used in
  `FACE_PARTS` and `STATE_FACE` (assert at import, or expose a checkable function the
  smoke-test/exercise can call), so a double-width glyph fails loudly.
- **Restyle the existing five states** (happy / ok / hungry / tired / sad): update
  `FACE_PARTS` (boxed eyes/mouth), `STATE_FACE` (the compact one-liner used in the intro
  "blinks awake" line), and `STATE_COLOR` (keep the semantic colors — green/cyan/yellow/red
  — adjust only if it reads better).
- **Add a `content` state**: a gentle "comfortable but not euphoric" tier. Extend the
  `face_state` cascade with a principled threshold (e.g. all needs comfortable but below
  the `happy` euphoric bar) and give it its own glyphs + color.
- Keep `render.py` pure and ANSI-degrading; no new dependency.
- Illustrative direction (final glyphs chosen + width-verified in this issue):
  ```
  content  ╭───────╮      happy  ╭───────╮
           │ ✶   ✶ │             │ ✦   ✦ │
           │  ‿‿‿  │             │  \_/  │   (or a width-safe smile)
           ╰───────╯             ╰───────╯
  ```

### Dependencies
None.

### Expected result
The creature shows a distinct, on-brand sparkle/minimal face set for six need-derived
states (happy / content / ok / hungry / tired / sad); every glyph is verified
single-width so the box stays aligned; the status layout and v0.1 in-place redraw are
untouched.

### Acceptance criteria
- [ ] The five existing states + `content` render in the new Claude-adjacent aesthetic and
      are visually distinct.
- [ ] A width guard (`unicodedata.east_asian_width`) asserts every face glyph is
      single-width (Narrow/Neutral/Halfwidth; `W`/`F`/`A` rejected).
- [ ] Box alignment holds for every state (7-wide interior; lines stay equal width).
- [ ] `STATE_FACE` compact faces are updated and width-safe (intro line renders cleanly).
- [ ] `status_line` layout and `STATUS_HEIGHT` are unchanged; the v0.1 in-place redraw
      still works (PTY/interactive unaffected; piped smoke-test clean).
- [ ] `--rule` and LLM paths render the same faces (rendering is brain-agnostic).
- [ ] `python -m py_compile creature/render.py` + import pass; smoke-test exits 0.
- [ ] Pure & stdlib-only; two-clock split intact (no change to needs/pet/brain).

## PIP-006: Thinking face + explicit-state render seam

### Description
Add a `thinking` face and a way to render an **explicit** state (not derived from
`Needs`), so v0.3 can show "pip is thinking" while an async reply is pending. v0.2
provides the capability only; the live wiring is v0.3.

### What needs to be done
- **Add a `thinking` state** to `FACE_PARTS` / `STATE_FACE` / `STATE_COLOR` in the
  PIP-005 aesthetic — e.g. sparkle `✻` eyes and a quiet `·` mouth, in a dim/cyan color —
  width-verified by PIP-005's guard.
- **Add an explicit-state override**: give `face_block`, `status_line` (and optionally
  `face`) an optional `state: str | None = None` parameter that uses the given state
  instead of `face_state(n)` when provided. **Default `None` preserves today's behavior**,
  so existing `chat.py` callers (`status_line(pet.name, pet.needs)`) are unaffected.
- `thinking` must **not** be reachable from `face_state(needs)` (it is a transient UI
  state) — it is shown only via the explicit override.
- **Do not wire it into `chat.py`.** The async "show thinking while the reply is in
  flight" belongs to v0.3; this issue is the render seam v0.3 will call.

### Dependencies
- PIP-005 (the new aesthetic/palette + the width guard).

### Expected result
`status_line(name, n, state="thinking")` (and `face_block(n, state="thinking")`) render
the thinking face with the box aligned, while the default no-argument calls behave exactly
as before.

### Acceptance criteria
- [ ] A `thinking` face exists and is rendered when an explicit `state="thinking"` is
      requested (via `face_block` / `status_line`).
- [ ] Omitting the override leaves all current behavior identical (chat.py callers
      unchanged; v0.1 redraw unaffected).
- [ ] `thinking` glyphs pass PIP-005's width guard; box stays aligned.
- [ ] `face_state(needs)` never returns `thinking` (it is override-only).
- [ ] Forward link noted: v0.3 calls the override while a reply is pending.
- [ ] `python -m py_compile creature/render.py` + import pass; smoke-test exits 0.
