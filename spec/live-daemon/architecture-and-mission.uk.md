# pip — Живий демон: Архітектура і місія

> **Статус:** проєктний документ, ще не реалізовано.
> **Обсяг:** перетворити pip із однопроцесного REPL на резидентний демон **pip-server** + живий **pip-client**, з **асинхронним розумом**, **чатом у каноні pip** та **памʼяттю сесії**.
> **Трек:** *Живий демон* — **v1** (фази `v1.0`–`v1.5`), самостійний, фундаментний трек, що є підмурівком для візії фронт-деска з [`../vision/pip-frontdesk-spec.md`](../vision/pip-frontdesk-spec.md) (**v2**, **фінальний цільовий стан**).
> **Дорожня карта:** глобальний, версійно-ключований план — [`../ROADMAP.md`](../ROADMAP.md) (англійською).
> **Обмеження:** перша версія лишається **навчальним проєктом** — лише стандартна бібліотека, один концепт на модуль, коментарі навчального рівня та документація-гайд. Див. [§9 Навчальні обмеження](#9-навчальні-обмеження).

## 1. Місія

pip — істота, що *живе* у твоєму терміналі. Щоб жити по-справжньому, вона має жити й тоді, **коли ти відвернувся** — зголодніти, занудьгувати, дрейфувати настроєм і подавати голос сама, — а не завмирати щойно ти перестав друкувати.

Сьогодні pip живе лише поки її REPL-цикл є процесом переднього плану. Тільки-но цикл блокується на вводі (чи на повільному виклику API), годинник істоти фактично спиняється, і немає способу *спостерігати*, як вона змінюється.

Цей трек дає pip три речі, потрібні, щоб бути живою:

1. **Власне серцебиття** — резидентний **демон** (`pip-server`), чиє тіло тікає в реальному часі безперервно, з під'єднаним клієнтом чи без нього.
2. **Вікно, яке можна відкрити** — живий **клієнт** (`pip-client`), що показує мордочку, настрій і потреби, які змінюються в реальному часі, поки ти розмовляєш.
3. **Розум, що думає не затамувавши подих** — **асинхронний мозок**: LLM запитуємо поза годинником тіла, тож повільна відповідь ніколи не спиняє серце.

Це водночас **підмурівок фінальної цілі**: [візія фронт-деска](../vision/pip-frontdesk-spec.md) — pip як концьєрж до флоту агентів — *передбачає* pip, яка завжди ввімкнена, завжди обізнана, мультиклієнтна та здатна говорити без запиту. Фронт-деск не може жити всередині блокувального REPL. **Перш ніж pip зможе наглядати за флотом, сама pip має стати резидентним демоном з живим вікном і асинхронним розумом. Це і є цей трек.**

Перша версія все одно лишається **навчальним проєктом**: демон, сокет і пул потоків подаються як *навчальний матеріал* — малі, читабельні, лише на стандартній бібліотеці, один концепт на файл, описані в гайді.

## 2. Принципи дизайну (незмінні + один новий)

Ідентичність проєкту — **дизайн двох годинників** (див. [`creature/__init__.py`](../../creature/__init__.py)): **ТІЛО** з чистої математики, що тікає на швидкому годиннику реального часу, і **РОЗУМ** (LLM), якого запитуємо лише коли є що сказати. Цей трек зберігає принцип і робить його *фізичним*, а тоді додає одне нове правило:

| Принцип | До | Після (цей трек) |
|---|---|---|
| **Два годинники** | тіло й розум в одному процесі, один цикл | годинник тіла працює всередині **сервера**; розум запитуємо з нього |
| **Тіло чисте й завжди живе** | тікає щопрохід REPL | тікає безперервно в демоні, навіть **без клієнта** |
| **Розум лише коли є що сказати** | викликається інлайн, **блокує** цикл | викликається **поза потоком**; ніколи не блокує тіло чи сокет |
| **НОВЕ — тіло однопотокове** | (н/д) | тіло (`Needs`, історія) мутується **лише головним потоком сервера**; воркери отримують незмінний знімок |

Останнє правило — хребет безпеки асинхронного дизайну: жодних локів на тілі, жодних гонок даних — див. [§5](#5-асинхронний-розум-вимога-3).

## 3. Модель процесів і транспорту

```
            ~/.pip/pip.sock  (Unix-сокет, JSON через перенесення рядка)
                   │
   ┌── pip-server (демон) ──────────────────┐        ┌── pip-client (живий) ─────┐
   │  володіє Pet = Needs(тіло)+Brain(розум)│  push  │  потік-читач:             │
   │                                        │ ─────▶ │    status / say / thinking│
   │  головний потік:                       │        │  головний потік:          │
   │    • годинник тіла (тік на реальн. dt) │        │    • ввід у cbreak         │
   │    • цикл сокета (select з таймаутом)   │ ◀───── │    • перемальовка кадру    │
   │    • застосувати готові відповіді       │  send  │      (мордочка+смуги+чат) │
   │  пул воркерів:                          │ chat/  └───────────────────────────┘
   │    • виклики LLM (асинх., поза годинн.)  │ act
   │  персистентність: ~/.pip/state.json     │        (може під'єднатись багато клієнтів)
   └────────────────────────────────────────┘
```

- **`pip-server`** — резидентний демон. Володіє єдиним `Pet`. Його **головний потік** веде і годинник тіла, і цикл сокета; малий **пул воркерів** виконує виклики LLM. Зберігає стан у `~/.pip/state.json`.
- **`pip-client`** — тонкий, без власного стану, живий термінал. Малює те, що пушить сервер, і передає твій ввід. Може під'єднатись багато клієнтів водночас (дивитись із кількох панелей).
- **Транспорт** — **Unix-сокет** на `~/.pip/pip.sock` (перевизначення через `PIP_SOCK`). Обрано, бо він двонапрямлений, лише локальний, не потребує TCP-порту і його тривіально `select()`-ити поряд із годинником тіла. Повідомлення — **JSON, розділений переносом рядка** — один крихітний, сумісний наперед контракт (див. [§4](#4-протокол-обміну)).

Навіщо взагалі демон? REPL переднього плану може робити *одне* з {чекати ввід, тікати тілом, викликати API} за раз. Істота, що має бути **живою й спостережуваною**, потребує всіх трьох водночас. Розділення тіла (сервер) і вікна (клієнт) та винесення API поза потік — мінімальна архітектура, що це дає, і саме та форма, яку потребує візія фронт-деска.

## 4. Протокол обміну

Один малий JSON-обʼєкт на рядок. Свідомо мінімальний і **сумісний наперед** — візія фронт-деска додає нові *типи* повідомлень у той самий канал, а не новий канал.

**сервер → клієнт**

| `t` | дані | значення |
|---|---|---|
| `hello` | `{name, brain}` | при під'єднанні (клієнт дізнається імʼя істоти й активний мозок) |
| `status` | `{levels:{hunger,energy,mood}, state}` | тіло змінилось — пушиться за каденцією **і** після кожної події |
| `say` | `{name, text}` | сказана репліка: відповідь у чаті, на турботу чи спонтанний поклик |
| `thinking` | `{}` | хід чату/турботи відправлено в розум (клієнт показує «…») |
| `bye` | `{}` | сервер вимикається |

**клієнт → сервер**

| `t` | дані | значення |
|---|---|---|
| `chat` | `{text}` | розмова з pip → маршрутизується до LLM у каноні ([§6](#6-чат-у-каноні-вимога-4)) |
| `act` | `{kind: feed\|play\|sleep}` | дія турботи; числовий ефект застосовується **негайно** |
| `cmd` | `{name: status\|help}` | `status` ініціює пуш; `help` клієнт обробляє локально |
| `shutdown` | `{}` | зупинити демон (опційно) |

Це **розширює** наявний контракт мозку `{say, mood}` (див. [`creature/brain.py`](../../creature/brain.py)): клієнт ніколи не бачить `mood` — його застосовує сервер до тіла. **Зарезервовано для візії (поки без поведінки):** кадр `fleet` (сервер→клієнт, статуси агентів) та дієслова `agents` / `dispatch` (клієнт→сервер). Названі тут, щоб протокол не довелося ламати згодом.

## 5. Асинхронний розум (вимога 3)

Тіло мусить тікати, поки LLM думає. Правило, що робить це безпечним: **тіло мутується лише головним потоком сервера.** Воркери його не торкаються; вони отримують незмінний знімок і повертають відповідь.

```
клієнт шле {chat,text}
        │
  головний потік:  broadcast {thinking}
                   ctx = pet.snapshot()         # (копія Needs, копія історії) — на головному потоці
                   pool.submit(pet.think, kind, text, ctx)
        │
  потік-воркер:  reply = brain.respond(...)     # повільний виклик API; торкається ЛИШЕ знімка
                 queue.put((kind, text, reply))      # потокобезпечна передача
        │
  головний потік (наступний тік):  reply = queue.get()
                                   say = pet.apply_reply(kind, text, reply)  # мутація mood+історії ТУТ
                                   broadcast {say}, {status}
```

- `Pet.snapshot()` копіює `Needs` (датаклас float-ів) і список історії **на головному потоці**, тож воркер читає заморожений зріз — без гонок.
- `Pet.think(kind, text, ctx)` чистий: викликає `brain.respond` на знімку й повертає `{say, mood}`.
- `Pet.apply_reply(...)` виконується назад **на головному потоці**: застосовує дельту `mood` до живого тіла й додає хід в історію.
- Доставка — `queue.Queue`, що осушується раз на тік; затримка ≤ таймаут циклу сокета (~0,15 с).
- **Збережені деградації** (з нинішнього дизайну, важливо лишити): `LLMBrain.respond` далі відкочується до `RuleBrain` за будь-якої помилки API; **спонтанні поклики лишаються шаблонними й миттєвими** — їх *не* маршрутизуємо через API. Асинхронний шлях лише для прямого чату та відповідей на турботу.

Підсумок: ти годуєш pip — і смуги рухаються *зараз*; ти щось кажеш — і смуги далі дрейфують, *поки* модель пише відповідь — істота ніколи не завмирає. Цей асинхронний шов потрібен і для наміру **dispatch** із візії (запуск/очікування агента — це просто повільніший «think»).

## 6. Чат у каноні й памʼять сесії (вимоги 4 і 5)

- **Чат у каноні (вим. 4):** повідомлення `chat` маршрутизується до `LLMBrain` із наявною персоною `SYSTEM` у [`creature/brain.py`](../../creature/brain.py) — мʼякий голос істотки, контракт JSON `{say, mood}` без змін. Без ключа API чи з `--rule` деградує до `RuleBrain` точно як тепер. Дії турботи (`feed`/`play`/`sleep`) теж отримують репліку в каноні тим самим асинхронним шляхом, *після* того, як ляже числовий ефект.
- **Памʼять сесії (вим. 5):** `Pet` накопичує ходи чату на **всю сесію сервера** (а не лише на один запуск REPL). Контекст LLM (`_build_context`) включає **вікно нещодавніх ходів**, тож pip памʼятає, що ти казав раніше в сесії. Оскільки сервер переживає будь-якого окремого клієнта, **памʼять переживає відʼєднання/перепідʼєднання клієнта** в межах сесії. Малий ліміт обмежує памʼять і токени; регулятор (`PIP_HISTORY`) налаштовує, скільки ходів доходить до моделі. (Тривала памʼять між сесіями свідомо *поза обсягом* тут — довготривале сховище у візії це журнал флоту, а не чат.)

## 7. Живий клієнт (вимоги 1 і 2)

Завдання клієнта: зробити зміну **видимою в реальному часі**, поки ти друкуєш.

- **Повторне використання рендеру:** клієнт відновлює `Needs` із надісланих `levels` і викликає наявні чисті рендери з [`creature/render.py`](../../creature/render.py) (`status_line`, `face`, `bar`). Рендеринг лишається в одному місці; дані лишаються крихітними.
- **Живий цикл:** фоновий **потік-читач** оновлює спільний стан UI (останній статус, лог чату, прапор «thinking») під локом; **головний потік** читає клавіатуру в режимі `cbreak` через `select`, веде буфер вводу й **перемальовує весь кадр ~10 разів/с**. Саме це дозволяє мордочці й смугам рухатись, поки твій недодрукований рядок лишається цілим.
- **Гігієна терміналу:** використовує альтернативний екранний буфер (зберігає скролбек) і відновлює `termios` на виході. ANSI деградує мʼяко, узгоджено з `render.py`. Лише Unix (`select`/`termios`), відповідно до наявного зауваження про платформу.
- **Регіони з місцем для росту:** `мордочка + смуги потреб` · `лог чату` · `рядок вводу`. Розкладка свідомо лишає місце для майбутньої **панелі флоту** (візія §8) як четвертого регіону.

З `--speed > 1` дрейф видно за секунди; на типовій швидкості це повільна, фонова зміна — саме відчуття тамагочі, тепер безперервне й спостережуване.

### Точки входу

```
python pip.py server            # демон переднього плану (Ctrl-C зупиняє)

```

## 9. Навчальні обмеження

Перша версія — спершу навчальний матеріал, потім софт. Непорушне:

- **Лише стандартна бібліотека.** Жодної нової залежності. `anthropic` лишається єдиною, опційною. Сокети, `select`, `threading`, `queue`, `termios`, `json` — усе stdlib.
- **Один концепт на модуль.** `protocol` = кадрування, `server` = цикл демона, `client` = живе вікно, `state` = персистентність. Жоден модуль не робить дві роботи.
- **Історія двох годинників лишається читабельною.** Демон має *прояснювати* поділ тіло/розум, а не розмивати його. Нове правило «тіло однопотокове» викладається явно, бо це суть безпечної асинхронності.
- **Коментарі пояснюють «чому».** Тримати щільність і тон наявних модулів (див. [`creature/chat.py`](../../creature/chat.py), [`creature/pet.py`](../../creature/pet.py)).
- **Документація йде з кодом.** Кожна фаза розширює [`GUIDE_uk.md`](../../GUIDE_uk.md) розділом (демон, протокол сокета, асинхронний розум, живий клієнт) і оновлює [`CLAUDE.md`](../../CLAUDE.md) та український [`README.md`](../../README.md). Екскурсія-гайд — частина результату, а не доважок.
- **Режим `solo` збережено** як найменший можливий навчальний шлях і офлайн/smoke-точка входу.

## 10. Ризики й відкриті рішення

| Тема | Нотатка / рішення, яке треба ухвалити |
|---|---|
| **Портативність** | `select` + `termios` + `cbreak` лише для Unix — уже заявлене обмеження проєкту (macOS/Linux). Без Windows. |
| **Один екземпляр** | виявляти наявний демон під'єднанням до сокета; чистити **застарілий** файл сокета при bind після збою. |
| **Спонтанне без клієнта** | коли ніхто не під'єднаний — **черга** поклика з виплеском при під'єднанні, чи **відкидання**? (Черга = добріше до тамагочі.) — *відкрите* |
| **Ріст історії / токени** | обмежити історію сесії; виставити `PIP_HISTORY` для вікна LLM. Тривала памʼять відкладена до журналу візії. |
| **Запис від кількох клієнтів** | багато клієнтів можуть слати ввід; сервер серіалізує все на головному потоці, тож це природно безпечно. |
| **Detach vs передній план** | типово `server` працює на передньому плані (легко вчити й Ctrl-C); `--detach` — шлях «справжнього демона» з логом. |

## 11. Звʼязок із фінальною ціллю (візія фронт-деска)

Цей трек — **фундамент Фази 0 + зерно Фази 6** з [`../vision/pip-frontdesk-spec.md`](../vision/pip-frontdesk-spec.md). Кожен елемент тут будується для повторного використання, не для переробки:

| Елемент Живого демона | Слугує фазі візії |
|---|---|
| `state.json` + старіння на час відсутності ([§6 тут]) | **Фаза 0** — *PIP CORE → state.json* |
| Шов асинхронного розуму ([§5]) | **Фаза 2/3** — концьєрж-**dispatch** це повільніший «think»; має бути поза годинником тіла |
| Протокол Unix-сокета ([§4]) | **Фаза 5** — **daemon-адаптер** (forage через сокет) перевикористовує цей транспорт |
| Резидентне, завжди-живе тіло ([§3]) | **Фаза 4** — проактивна **обізнаність про флот** потребує процесу, що наглядає, поки ти простоюєш |
| Регіони живого клієнта ([§7]) | **Фаза 6** — **панель флоту** стає поряд із мордочкою й потребами |
| Зарезервовані повідомлення `fleet`/`agents`/`dispatch` ([§4]) | **Фаза 1–3** — дієслова реєстру й диспатчу лягають у той самий канал |

Одним рядком: **зробити pip *живою* сама по собі, спостережуваною й такою, що думає без зупинок — щоб згодом вона стала фронт-деском до цілого флоту.**
